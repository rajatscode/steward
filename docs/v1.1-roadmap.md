## Tier 1 — Real frictions surfaced overnight that didn't ship

These are the items most likely to change the user's day-to-day in week 1. Pick one of these first.

### 1a. Apple HealthKit read-only (sleep / weight / steps)

Biggest leverage for the most likely first domain (Health). PM proposed this overnight; deferred for time risk. Reading sleep duration, weight (latest sample), and step count daily means the user doesn't have to manually log values their phone already knows.

**Scope:** ~2h.

**Files a HealthKit reader would touch:**
- New: `ios/Steward/HealthKit/HealthKitGateway.swift` (mirrors `ios/Steward/EventKit/EventKitGateway.swift` shape — permission, typed read of `HKQuantityType.sleepAnalysis` / `.bodyMass` / `.stepCount`).
- New tool: `ios/Steward/Tools/Catalog/HealthKitTools.swift` exposing `health.read_sleep(date)`, `health.read_weight()`, `health.read_steps(date)`.
- Wire into `ios/Steward/Tools/ToolCatalog.swift` registration.
- Permission flow: `ios/Steward/Views/Root/` onboarding sequence picks up a new step between notifications and EventKit. Add an `Info.plist` `NSHealthShareUsageDescription` entry.
- Auto-ingest path: `ios/Steward/Background/BGTaskCoordinator.swift` gets a `MorningHealthIngestJob` that runs at brief-time-minus-15min, reads yesterday's sleep + weight + steps if a Health domain exists with matching instruments, and emits `instrument.apply_event` events with `source='healthkit'`.

**Out of scope:** writing back to HealthKit (Tier 3), heart rate / HRV / workouts (later passes).

### 1b. Custom instrument kinds (user-defined `apply` via sandboxed expression eval)

`spec.md` §6 lists 7 built-in kinds. The user can already spawn instruments of those 7 via chat (e.g., "track my sleep as a rolling average") but cannot define their own update function. Real product gap: e.g. "track my caffeine cutoff time — first coffee timestamp each day, and warn me if it's after 2pm."

**Scope:** ~3-4h. Needs care on the sandbox.

**Approach:**
- Add an `eighth` kind in `ios/Steward/Instruments/InstrumentKind.swift` called `.custom`, with a `definition_json` holding a small expression in a restricted DSL (no loops, no I/O, only arithmetic + comparisons + the previous state).
- New evaluator in `ios/Steward/Instruments/Kinds/CustomExpression.swift`. Use a hand-rolled recursive-descent parser, not JavaScriptCore (avoids the security + size cost of bringing in a full interpreter).
- Coordinator can write the expression for the user from a chat description ("daily count, reset midnight, alert when > 3"); user can edit it in Settings → Teams → instrument detail.
- New tool: `instrument.create_custom(name, domain, definition_expression, capture_prompt)`.

**Open design questions** before starting:
- How does the LLM know when to propose `.custom` vs. one of the seven built-ins? Probably: try built-ins first, fall through to `.custom` only when no built-in fits.
- What's the error surface when the user's expression doesn't parse? Don't let it crash the agent loop; return a typed error the coordinator can read.

### 1c. Settings UI mutations not audit-logged

Today's mercy / pause / quiet-hours toggles in `ios/Steward/Views/Settings/ModesSection.swift` and `TimingSection.swift` call `SettingsStore.update(...)` directly. Settings mutations made via chat tools (`mercy_mode.engage`, `quiet_hours.set` in `Tools/Catalog/SettingsTools.swift`) emit events with `actor='coordinator'`. UI mutations should emit equivalent events with `actor='user'` and `kind='settings_change'`.

**Scope:** ~30 min. Smallest of the Tier 1 items; could be a warm-up before tackling 1a or 1b.

**Files:**
- `ios/Steward/Views/Settings/SettingsViewModel.swift` — route every mutation through a single helper that also emits an event.
- `ios/Steward/Tools/Catalog/SettingsTools.swift` — already emits events; align the payload shape between user-origin and agent-origin events so the audit log treats them uniformly.
- New test in `ios/StewardTests/SettingsStoreTests.swift` covering the event-emission path.

**Why this matters:** the audit log in Settings → Recent agent actions is sold to the user as "every external mutation is logged with a reason and an undo button." Today the audit story has a hole the user can step in.

## Tier 2 — Polish that becomes valuable in week 2+

These don't move the needle on first-week usefulness but they're real upgrades once the user has been logging for a few days.

### 2a. Weekly review report (cross-domain pattern recognition)

`spec.md` §16 and §21 reference this as a v1.1 deferral. The coordinator can do it ad-hoc on request today; structured weekly report needs design + dogfooding.

**What it would do:** Sunday-evening generated report card across all active domains — what trended up, what trended down, what got logged a lot, what went quiet. No moralization. Surfaces patterns ("you bed-rotted on the same day you skipped breakfast for three weeks in a row") without prescribing.

**Pre-req:** at least 2 weeks of real usage data so the report has something to recognize.

### 2b. Long-press chat context menus

Designer spec §1.2: Copy bubble text, View timestamp, Re-run agent from this point. The chat bubble views render today; the long-press affordances have no test coverage and weren't exercised in the simulator.

**Files:** `ios/Steward/Views/Chat/ChatView.swift`, ChatBubble views, ChatViewModel.

**Why Tier 2:** nice once you're using the app daily, missing on day 1 is invisible.

### 2c. Memory decay accrual watch

`v0.9.2` shipped the wiring: `MemoryDecayJob` is now invoked from `BGTaskCoordinator` and on app launch. Tier 3 watch item: confirm over a real week of data that strength values decay as designed (0.995/day base, type modifiers per `spec.md` §9), boost-on-retrieval works, and soft-delete at strength < 0.05 triggers correctly. No code change is the goal — just verify the wiring under real conditions.

**Action item if drift is seen:** the decay constants live in `ios/Steward/Memory/MemoryDecayJob.swift` and are individually unit-tested.

## Tier 3 — Engineering not user-visible

Real work, but the user wouldn't notice if it shipped tomorrow vs. in a month.

- **CloudKit multi-device sync.** `spec.md` §21 → v2. The current single-device design is load-bearing for simplicity; adding CloudKit means deciding on conflict resolution for the event log (last-writer-wins on instruments is fine; on events nothing should ever overwrite).
- **Tiny webhook/push relay for true Calendar-pushed wake-ups.** `spec.md` §10 and Researcher landmine: iOS BGTasks are heuristic. Pre-scheduled notifications cover the planned cases; a hosted relay would let calendar invites trigger wake-up logic in (close to) real time. Cost: a small backend. Push notifications from a server are otherwise on the anti-list below — this is the one narrow case where they're justified.
- **HealthKit write-back.** Separate from Tier 1a. Write Steward-logged events back to Health when a corresponding HK quantity exists (e.g., user logs weight in Steward → write to `HKQuantityType.bodyMass`). Adds a permission scope; defer until Tier 1a has bedded in.
- **Google Sheets pluggable target.** `sync_queue.target` field already supports adding `'sheets'`. Only matters for interpretation B from the spreadsheets discussion (user-writable freeform formulas). User accepted in v1 spec discussion that interpretation A (agent-maintained auto-computation) is enough. Build when first useful.
- **Plaid bank sync for Money domain.** High value if a Money domain takes off, but real engineering + credentials risk. v2.
- **Mac companion app.** `spec.md` §21 → v2. iPhone is canonical. CloudKit sync would enable it later.
- **Apple Foundation Models migration when iOS 26 SDK leaves beta.** Currently `LLMResolver` has a `MockLLMSession` fallback path with a typed `STUB` chip. When the SDK ships stable and the deployment target is updated, the fallback path can be removed entirely (and the `MockLLMSession` retained only as a test double).

## What we will NOT do

Anti-bloat list. Listed here so future-Rajat doesn't have to re-litigate.

- **Streak counters.** `spec.md` §2 principle 1 (continuity over intensity) + §15 anti-moralization. Streaks are a streak-reset shame engine; the spec explicitly names this an anti-goal.
- **Push notifications from a server.** Offline-first is load-bearing. The one narrow exception (Calendar-pushed wake-up via webhook relay, Tier 3) is the only kind of server-originated push we'd consider, and only for a wake-up trigger, never for content.
- **Multi-user.** Single user is load-bearing. `spec.md` §0 / §1 / §3. Adding multi-user means re-thinking the memory layer (whose lessons?), the audit log (whose reasoning?), and the agent loop (whose role prompt?) — none of which the current design accommodates.
- **A more aesthetic to-do list.** Listed in `spec.md` §1 anti-goals. The product wins on continuity, not on UI flourish.
- **A quantified-self dashboard the user services rather than uses.** Listed in `spec.md` §1 anti-goals. If the user finds themselves staring at instrument cards and not logging, that's a failure mode to design against, not toward.
- **A moralizing coach.** Listed in `spec.md` §1 anti-goals. Reiterated in `spec.md` §15 anti-moralization clauses. Hard-coded into both coordinator and domain agent system prompts.
- **A streak-reset shame engine.** Listed in `spec.md` §1 anti-goals. Reiterated as banned pattern in `design/coordinator-empty-state-v2.md` §8.

## How to pick next

Read this file. Pick one Tier 1 item. Spawn a `/hackathon` with that scope.

Don't pick more than one in a session. v0.9 → v1.0 happens via accumulated small wins, not big bangs. Each Tier 1 item is sized so a single overnight pass can land it, ship a tagged build, and leave time for the next morning's self-QA. If you find yourself wanting to bundle two items "because they're related," that's the signal to do one this session and the other next.

If none of Tier 1 feels right (e.g., you haven't used the app for a week and don't yet know which friction is biggest), default to **1c** (Settings UI audit log). It's the smallest, it closes an audit-story hole the user might notice on Sunday, and it doesn't commit you to a direction.
